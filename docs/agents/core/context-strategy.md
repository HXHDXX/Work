# 上下文与沙盒策略 (Context-Mode)

> AGENTS.md「上下文与沙盒策略」清单的详细说明。

## 背景

Agent 的上下文窗口有限。大型项目中，原始数据（构建日志、大文件、网络响应）会迅速消耗上下文。Context-Mode 通过分层信息处理，将上下文占用压缩到 2%（98% 压缩率）。

## 核心机制

### 默认粗筛

- **优先使用** `ctx_search` / `ctx_execute_file` 获取结构化摘要
- 这些工具返回经过筛选的关键信息，而非原始数据
- 适用于 98% 的日常开发场景

### 按需放行

以下场景需要原始数据，通过钩子请求：

| 场景 | 原因 | 示例 |
|------|------|------|
| 内存安全 | 需要完整内存布局、地址信息 | ASAN 报告、valgrind 输出 |
| 跨进程寻址 | 需要精确地址和偏移量 | gdb 远程调试、core dump 分析 |
| 构建环境修改 | 需要完整的环境变量和路径 | CMake 配置、Docker 构建参数 |

### 决策对焦

关键决策点需询问用户：

```
基于摘要判断，还是查看原始数据?
```

- 摘要足够时 → 继续使用摘要
- 摘要不够时 → 放行原始数据，但限定范围

### 沙盒强制

所有非交互式执行必须路由至 `ctx_execute`：

| 操作类型 | 必须沙盒 |
|----------|---------|
| 非交互式脚本 | 是 |
| 网络请求 | 是 |
| 大文件扫描 | 是 |
| 交互式命令 | 否（直接执行） |

### 指令拦截

以下原生命令被禁用，由 MCP Hooks 重定向：

| 原生命令 | 重定向到 | 原因 |
|----------|---------|------|
| `curl` | `ctx_execute` | 防止大量网络响应污染上下文 |
| `find` | `ctx_search` | 防止大目录树遍历输出 |
| `grep` | `ctx_search` | 统一搜索接口，返回结构化结果 |

## 操作流程

```
收到任务
  ↓
ctx_search 检索相关代码摘要
  ↓
摘要足够? ── 是 ──→ 执行任务
  ↓ 否
询问用户: 摘要 or 原始数据?
  ↓
用户选择摘要 → 尝试更精确的 ctx_search
用户选择原始 → ctx_execute 放行（限定范围）
  ↓
执行任务
```

## 注意事项

- 沙盒不是安全边界，是上下文管理工具
- 如果 `ctx_search` / `ctx_execute` 不可用（非 MCP 环境），退化为直接工具调用
- 记录每次放行原因到知识库，积累模式识别经验
