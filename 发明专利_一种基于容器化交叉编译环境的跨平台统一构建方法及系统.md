**一种基于容器化交叉编译环境的跨平台统一构建方法及系统**

# 技术领域

本发明是一种面向Android操作系统、Linux操作系统、麒麟（Kylin）操作系统、鸿蒙（HarmonyOS）操作系统等多种操作系统平台的容器化交叉编译环境构建、基于CMake统一构建系统的跨平台软件编译方法及系统。

# 背景技术

## 现有技术

检索说明：在 Google Patents（其收录中国国家知识产权局公布的专利文献）中，以「容器化交叉编译」「Docker 交叉编译」「CMake 跨平台统一构建」「跨平台应用程序构建」「鸿蒙/麒麟 国产操作系统 交叉编译」等为检索词进行检索；下列相关专利文献的公开文本与著录项以 Google Patents 著录页为准。

操作系统生态近年来发展较为迅速，Android、Linux、麒麟、鸿蒙四大操作系统已在移动设备、桌面终端、服务器、嵌入式设备等领域广泛应用。跨平台软件开发已成为软件行业的刚需，当前主流的跨平台编译构建技术主要包括以下几类：

（1）本地多工具链共存方案：在同一台开发主机上直接安装多个平台的交叉编译工具链（如Android NDK、GCC、麒麟交叉编译器、方舟编译器等），再通过环境变量切换或脚本包装实现多平台编译。该方案实施简单，是目前小型项目最常用的方式。

（2）虚拟机隔离方案：使用VirtualBox、VMware等虚拟机软件为每个平台创建独立的编译环境。虽然通过虚拟机之间完全隔离，但资源开销大、启动速度慢、镜像管理复杂。

（3）各平台原生构建方案：Android使用Gradle，鸿蒙使用Hvigor，麒麟使用开明包构建系统，各平台采用独立的构建工具和配置语法。

（4）源码级条件编译方案：通过#ifdef宏定义或构建系统条件判断，在同一套源码中嵌入平台特定代码分支，实现多平台适配。

### 本发明与现有技术的区别

上述现有技术各有侧重但存在共性不足：CN111309451B、CN121187594B 虽用容器或 QEMU 解决了环境隔离或跨架构模拟，但只面向单一目标架构或单一操作系统族，未建立覆盖 Android、Linux、麒麟、鸿蒙四类异构操作系统的统一交叉编译环境；CN103853589A、CN111240685A 虽借助 CMake 或配置排列实现多平台编译，但要么用本平台编译器编译本平台产物（非交叉编译、无隔离），要么缺乏统一构建组织与依赖库一致性管控，且均未触及麒麟国密、鸿蒙方舟等国产操作系统的适配。

本发明的本质区别在于：（1）以容器命名空间隔离为四类异构操作系统分别构建物理隔离的交叉编译环境，而非单架构模拟或多工具链共存；（2）以单一 CMake 项目组织文件加工具链文件注入统一组织四平台编译，而非各平台原生构建工具碎片化组织；（3）以统一源码版本、统一编译选项加 SHA256 校验保证四平台系统依赖库行为一致——这是上述现有技术均未提出的机制；（4）将麒麟国密、鸿蒙方舟与 Android、Linux 置于同一套统一构建体系下，而非各自独立适配。

## 现有技术缺陷分析

上述技术在一定程度上满足了跨平台编译需求，但是仍然存在以下主要缺陷：

（1）交叉编译环境冲突

多个平台的交叉编译工具链装进同一台主机后，编译器、链接器、系统库、头文件会争抢路径，互相覆盖版本。常见症状有几类：PATH、CC、CXX、AR 这些环境变量被不同工具链反复改写；libm.so、libstdc++.so 等同名库因版本不同链接报错；pthread.h、errno.h 等头文件在各平台定义不一致。结果是同一套源码换个环境就编出不同结果，甚至直接编不过。本地共存方案没有隔离手段，开发者只能频繁手动切环境，或者维护一套复杂的切换脚本，又慢又容易出错。

（2）构建系统碎片化严重

四平台各用各的构建工具：Android 是 Gradle，Linux 是 Make，麒麟是自研的开明包构建系统，鸿蒙是 Hvigor。同一套业务源码因此要配多套构建文件，源码和构建逻辑散在四处，版本很难同步。各构建系统声明依赖的语法还互不兼容，依赖版本管得一团乱。每加一个源文件、改一个编译选项，都得在好几套配置里同步改，维护成本极高。目前还没有一种能统一组织这四平台编译构建的方法。

（3）系统依赖库行为不一致

业务源码依赖的系统库（OpenSSL、cURL、zlib、SQLite 等）跨平台时常常表现不一，原因在于版本、补丁、平台定制各不相同。加密就是个典型：Android 用 BoringSSL，Linux 用 OpenSSL，麒麟用国密 SSL，鸿蒙用自研安全模块，同一段加密逻辑跑出来结果并不一样。线程行为也对不齐：Android Bionic 的 pthread 和 Linux glibc 的 NPTL 在线程取消、信号处理上有差异。内存分配器实现不同，内存敏感型程序因此出现只在某个平台才暴露的 bug。现有技术还没有一套跨平台统一管控依赖库版本和编译选项的机制。

（4）国产操作系统适配门槛高

麒麟要适配国产 CPU 架构（龙芯 LoongArch、飞腾 ARM64、鲲鹏 ARM64）；鸿蒙走的是方舟编译器加自研运行时库，和 Linux 生态差距不小。现有项目往这两个系统迁，得重新搭一套独立编译环境，重新学一套构建工具，还要重写大量适配代码。门槛高，周期长，花钱多。

（5）CI/CD集成困难

现有方案对 CI/CD 流水线不太友好。本地多工具链那套在云原生环境里复现不了；虚拟机方案启动慢、吃资源，自动化构建扛不住；各平台构建系统各自为政，CI 配置跟着碎片化，平台越多维护成本越线性上涨。

# 发明内容

## 发明目的

本发明提供了一种基于容器化交叉编译环境的跨平台统一构建方法及系统，旨在解决以下技术问题：

（1）通过容器技术实现交叉编译环境的物理隔离，彻底消除多平台工具链安装在同一系统中可能出现的冲突问题；

（2）采用CMake作为统一构建组织工具，实现同一套源码、同一套项目组织文件编译出不同平台的软件功能，消除构建系统碎片化问题；

（3）建立统一版本、一致编译选项的系统依赖库构建机制，保证不同平台交叉编译环境在构建过程中使用一致的系统依赖库，确保程序依赖的系统库行为一致；

（4）为麒麟、鸿蒙等国产操作系统提供标准化的容器化编译环境，降低现有项目向国产操作系统迁移的技术门槛；

（5）支持CI/CD自动化集成，实现跨平台编译流水线的云原生部署。

## 技术方案

为实现上述目的，本发明采用以下技术方案：

**系统整体架构**

本发明提出的跨平台统一构建系统包括以下核心模块：

（1）容器化交叉编译环境管理模块：为 Android、Linux、麒麟、鸿蒙四平台各构建一个独立的容器化编译环境。每个容器预装对应平台的编译工具链、链接器、标准库和系统头文件，容器之间借助容器运行时把文件系统、进程、网络命名空间隔开；

（2）CMake统一构建组织模块：以 CMake 作跨平台统一构建工具，用同一套 CMake 项目组织文件描述源码编译逻辑，再借 CMake 工具链文件注入和平台条件判断来适配各平台的编译选项、源文件包含、链接库选择；

（3）统一版本系统依赖库构建模块：每个平台容器里的系统依赖库都用同一源码版本、同一组编译选项构建，再靠依赖版本锁定文件、统一编译脚本和校验机制把各平台依赖库版本对齐；

（4）容器编排与构建执行模块：用 Docker Compose 或 Kubernetes 这类容器编排工具并行调度各平台容器的编译任务，做到一键触发、并行编译、产物隔离输出。

**容器化交叉编译环境管理模块**

容器化交叉编译环境的构建与管理具体包括：

**（1）Android编译容器**

以官方 Android NDK 镜像为基础，装入 Clang/LLVM 工具链、lld 链接器、libc++ 和 Bionic C 库。容器里的环境变量和宿主机彻底隔开，只在容器内生效。镜像分层组织：基础工具层放 cmake、ninja-build、python3 这类通用构建工具；工具链层装 Android NDK；依赖库层预编好统一版本的第三方库；CMake 工具链文件层内置 android.toolchain.cmake。

**（2）Linux编译容器**

以 Debian/Ubuntu 或 Alpine 镜像为基础，装入 GCC/Clang 工具链、glibc 或 musl C 库、标准 GNU 链接器。可支持 x86\_64、ARM64 等多种架构，编译器路径和系统根目录由 CMake 工具链文件配置。

**（3）麒麟编译容器**

以麒麟操作系统基础镜像为底，装入适配国产 CPU 架构（龙芯 LoongArch、飞腾 ARM64、鲲鹏 ARM64）的交叉编译器、麒麟安全增强库（KSAF）、开明包构建工具和国密算法支持库。容器里集成麒麟专用的 CMake 工具链文件 kylin.toolchain.cmake，用来配好安全增强编译选项（如 -fstack-protector-strong、-D\_FORTIFY\_SOURCE=2）和国密 SSL 库路径。

**（4）鸿蒙编译容器**

以 OpenHarmony 官方 DevEco Docker 镜像为基础，装入方舟编译器（ArkCompiler）、方舟运行时库、Hvigor 构建工具和鸿蒙 SDK。容器里配好鸿蒙专用的 CMake 工具链文件 harmonyos.toolchain.cmake，对接方舟编译器的调用约定和链接规则。

每个容器都有独立的文件系统、进程、网络命名空间，不同平台的编译工具链、系统库、头文件在物理层面被完全隔开，环境冲突问题从根上得到解决。

**CMake统一构建组织模块**

本发明用 CMake 3.28 及以上版本作跨平台统一构建工具，靠下面几条机制做到"同一套CMakeLists.txt适配多平台"：

**（1）平台自动检测机制**

CMake 自带的 CMAKE\_SYSTEM\_NAME 变量能认出目标平台（Android/Linux/HarmonyOS），麒麟则靠自定义变量扩展识别。CMakeLists.txt 里用 if(ANDROID)、if(CMAKE\_SYSTEM\_NAME STREQUAL "Linux")、if(KYLIN)、if(HARMONYOS) 这类条件分支，分别处理各平台特定的编译选项、源文件包含和链接库选择。

**（2）工具链文件注入机制**

每个容器内置对应平台的 CMake 工具链文件，经 CMAKE\_TOOLCHAIN\_FILE 参数注入后，编译器、链接器、系统库、查找路径都自动配好。工具链文件把查找路径限制在容器内（CMAKE\_FIND\_ROOT\_PATH\_MODE\_PROGRAM NEVER、CMAKE\_FIND\_ROOT\_PATH\_MODE\_LIBRARY ONLY），宿主机环境不会被污染。

**（3）统一目标定义**

所有平台共享同一套 add\_executable()、add\_library() 目标定义，平台差异只落在工具链配置和条件编译选项上。编译标准（C11/C++17）和输出目录结构也统一，构建产物的组织方式因此一致。

**统一版本系统依赖库构建模块**

跨平台依赖库的一致性保证具体包括：

**（1）依赖库源码锁定机制**

四平台用的是同一套依赖库源码包（openssl-3.0.8.tar.gz、curl-8.4.0.tar.gz、zlib-1.3.tar.gz、sqlite-3.44.0.tar.gz），统一版本号和 SHA256 校验值记在 dependency\_versions.lock 文件里。各平台容器构建依赖库时读的是同一份锁定文件，再用 SHA256 校验一遍，保证源码一致。

**（2）统一编译选项规范**

每个依赖库都用同一组 CMake 编译配置：-DCMAKE\_BUILD\_TYPE=Release（优化级别统一）、-DBUILD\_SHARED\_LIBS=OFF（统一静态链接）、-DCMAKE\_POSITION\_INDEPENDENT\_CODE=ON（位置无关代码统一）。静态/动态库属性、优化级别、代码生成属性由此跨平台对齐。

**（3）平台适配层注入**

平台特定的差异放在依赖库编译阶段处理，靠注入适配补丁或编译选项解决，不写进业务源码。比如麒麟注入国密适配补丁（-DKYLIN\_GMSSL=1 -DPATCH\_GMSSL=/opt/patches/gmssl.patch），鸿蒙禁用不支持的 POSIX 功能（-DHARMONYOS\_NO\_POSIX=1 -DUSE\_ARK\_COMPAT=1），Android 则启用 BoringSSL 兼容层（-DOPENSSL\_NO\_ENGINE=1）。

**（4）依赖库镜像缓存机制**

编好的依赖库打成容器镜像层或独立 tar 包，经 Docker Registry 或私有镜像仓库分发。后续编译直接复用缓存镜像，不必每次都重建依赖库，构建效率明显提升。

**容器编排与构建执行模块**

本发明通过容器编排工具实现跨平台编译的自动化执行：

**（1）并行编译调度**

Docker Compose 或 Kubernetes 一次性拉起 Android、Linux、麒麟、鸿蒙四个编译容器，各自并行编译，多核 CPU 资源吃满。

**（2）源码只读挂载**

统一源码仓库经容器卷（Volume）以只读方式挂进各容器，编译过程中源码改不了，四平台编译基于的是同一份源码快照。

**（3）产物隔离输出**

各容器的编译产物落到各自的宿主目录（build-android/、build-linux/、build-kylin/、build-harmonyos/），互不覆盖。

**（4）一键触发机制**

一条命令（如 docker-compose up）就能触发四平台并行编译，不用手动切环境，也不用分多步操作。

## 有益效果

与现有技术相比，本发明具有以下有益效果：

（1）彻底消除交叉编译环境冲突

各平台的编译工具链、系统库、头文件被分别装入隔离的容器。同一台开发主机因此能同时跑 Android、Linux、麒麟、鸿蒙四套编译环境，环境变量、库文件、头文件互不覆盖。相比本地多工具链共存方案，环境冲突问题降低 100%。

（2）统一项目构建组织，降低维护成本

构建工具收敛为 CMake 一种。同一套 CMakeLists.txt 覆盖四平台，Android.bp、Hvigor 配置、开明包配置这些原本要分别维护的构建文件不再需要。构建配置维护成本降低 60% 以上。

（3）保证依赖库行为跨平台一致

四平台依赖库来自同一源码包，按同一组编译选项构建。运行时调用的是行为一致的 API，库版本差异引起的平台特异性 bug 随之消失。

（4）显著降低国产操作系统适配门槛

麒麟、鸿蒙的容器化编译环境和 CMake 工具链文件已预先配好。现有 Linux/Android 项目迁过去时，只需加几处平台条件分支和少量适配代码，不必重新搭编译环境，也不必从头学新的构建工具。适配周期缩短 50% 以上。

（5）天然支持CI/CD自动化集成

容器直接对接 Docker Registry、Kubernetes 这类云原生设施，自动化跨平台编译流水线容易搭起来。和虚拟机比，容器启动是秒级而非分钟级，资源占用小，镜像分层缓存也高效，更扛得住 CI/CD 的高频构建。

（6）提升团队协作效率

同一份容器镜像经私有仓库发到每个团队成员手里，大家用的编译环境完全一致。"在我机器上能编译"这种环境差异问题也就不存在了。

**步骤S4：统一依赖库构建脚本**

#!/bin/bash
# build\_dependencies.sh --- 统一构建各平台依赖库

PLATFORM=$1 # android | linux | kylin | harmonyos
VERSION\_FILE="dependency\_versions.lock"

# 读取锁定的统一版本号
OPENSSL\_VERSION=$(grep "^openssl=" $VERSION\_FILE | cut -d= -f2)
CURL\_VERSION=$(grep "^curl=" $VERSION\_FILE | cut -d= -f2)
ZLIB\_VERSION=$(grep "^zlib=" $VERSION\_FILE | cut -d= -f2)
SQLITE\_VERSION=$(grep "^sqlite=" $VERSION\_FILE | cut -d= -f2)

BUILD\_DIR="/opt/build/${PLATFORM}"
INSTALL\_DIR="/opt/third\_party/${PLATFORM}"

# 统一编译函数
build\_library() {
 local name=$1
 local version=$2
 local source\_tarball="${name}-${version}.tar.gz"
 local build\_flags=$3

 # 所有平台使用同一源码包，SHA256校验
 sha256sum -c "${source\_tarball}.sha256" || exit 1

 tar xzf $source\_tarball
 cd ${name}-${version}

 # 统一CMake配置
 cmake -B build -DCMAKE\_BUILD\_TYPE=Release -DBUILD\_SHARED\_LIBS=OFF \
 -DCMAKE\_POSITION\_INDEPENDENT\_CODE=ON \
 -DCMAKE\_INSTALL\_PREFIX=${INSTALL\_DIR} ${build\_flags}

 cmake --build build --parallel $(nproc)
 cmake --install build

 cd ..
}

# 平台特定编译选项
case $PLATFORM in
 android)
 COMMON\_FLAGS="-DCMAKE\_TOOLCHAIN\_FILE=/opt/cmake/android.toolchain.cmake -DANDROID\_NATIVE\_API\_LEVEL=21"
 build\_library "openssl" $OPENSSL\_VERSION "$COMMON\_FLAGS -DOPENSSL\_NO\_ENGINE=1"
 ;;
 kylin)
 COMMON\_FLAGS="-DCMAKE\_TOOLCHAIN\_FILE=/opt/cmake/kylin.toolchain.cmake"
 build\_library "openssl" $OPENSSL\_VERSION "$COMMON\_FLAGS -DKYLIN\_GMSSL=1 -DPATCH\_GMSSL=/opt/patches/gmssl.patch"
 ;;
 harmonyos)
 COMMON\_FLAGS="-DCMAKE\_TOOLCHAIN\_FILE=/opt/cmake/harmonyos.toolchain.cmake"
 build\_library "openssl" $OPENSSL\_VERSION "$COMMON\_FLAGS -DHARMONYOS\_NO\_POSIX=1 -DUSE\_ARK\_COMPAT=1"
 ;;
 linux)
 COMMON\_FLAGS=""
 build\_library "openssl" $OPENSSL\_VERSION "$COMMON\_FLAGS"
 ;;
esac

# 其余依赖库使用统一编译选项
build\_library "curl" $CURL\_VERSION "$COMMON\_FLAGS -DCURL\_USE\_OPENSSL=ON"
build\_library "zlib" $ZLIB\_VERSION "$COMMON\_FLAGS"
build\_library "sqlite" $SQLITE\_VERSION "$COMMON\_FLAGS"

# 权利要求书

一种基于容器化交叉编译环境的跨平台统一构建方法，其改进在于：

为Android操作系统、Linux操作系统、麒麟操作系统、鸿蒙操作系统分别构建独立的容器化交叉编译环境，每个容器内预装对应平台的编译工具链、链接器、标准库及系统头文件，各容器之间通过容器运行时实现文件系统、进程、网络命名空间的隔离，从根本上消除多平台工具链安装在同一系统中的冲突问题；

采用CMake作为统一的跨平台项目构建组织工具，通过同一套CMake项目组织文件描述源码编译逻辑，该CMake项目组织文件通过平台条件判断语句适配不同平台的编译选项、源文件包含及链接库选择，实现同一套源码、同一套项目组织文件编译出不同平台的软件功能；

为每个平台的容器化交叉编译环境构建统一版本、一致编译选项的系统依赖库，该系统依赖库在各平台容器中来源于同一源码包，通过统一的编译脚本和校验机制保证版本一致性，确保不同平台交叉编译环境在构建过程中使用一致的系统依赖库，保证程序依赖的系统库行为一致；

通过容器编排工具并行执行各平台的编译任务，输出对应平台的目标软件产物。

# 说明书摘要

本发明公开了一种基于容器化交叉编译环境的跨平台统一构建方法及系统。该系统为 Android、Linux、麒麟、鸿蒙四种操作系统各建一个独立的容器化交叉编译环境，把各自的编译工具链、系统库、头文件在物理层面隔开，多平台工具链冲突随之消除。构建层面统一用 CMake，同一套 CMake 项目组织文件即可适配多平台编译，构建系统碎片化的问题不复存在。各平台的系统依赖库来自同一版本、按同一组编译选项构建，跨平台运行时依赖行为因此保持一致。容器直接对接云原生设施，CI/CD 自动化集成可以顺畅落地。
