## 三、这事儿是怎么来的（写给代理人：先了解我们要解决什么）

我们团队做的是跨平台基础软件，一套 SDK 得同时跑在四个操作系统上：Android（移动端）、Linux（桌面和服务器）、麒麟 Kylin（国产服务器，主要是信创项目）、鸿蒙 HarmonyOS（国产移动和嵌入式）。

做过跨平台编译的人都知道，这四个系统的编译环境根本不是一回事。最让我们头疼的不是写业务代码，而是"怎么把这四个平台的环境搭起来、还能让一套源码编出四个平台的产物、并且各平台依赖库的行为还对得上"。

我们踩了很多坑，试过本地装多套工具链、试过虚拟机隔离、试过各平台原生构建工具各搞各的，都不理想（下面会具体讲）。最后摸索出来的这套办法，核心就三句话：

- 用容器把四个平台的交叉编译环境**物理隔开**，谁也别干扰谁；
- 用 CMake 这一个构建工具**统一组织**四个平台的编译，别再用四套构建系统；
- 四个平台的系统依赖库**同一个版本、同一组编译选项**来构建，保证运行时行为一致。

这套办法我们已经在金融级加密 SDK、工业控制软件这些实际项目上跑通了，效果比传统方案好很多。我们觉得里面的几个机制是有创新性的，所以想申请专利保护。

## 四、在我们之前的那些方案，都有什么毛病

这块请代理人重点看，因为"现有技术的缺陷"是写背景技术时要靠它立论的。我们检索了 Google Patents 上国内已公开的相关专利，也结合自己踩坑的经验，把现有方案分成几类来说。

### 4.1 我们试过的几类老办法

**（一）本地多工具链共存**

就是在一台开发机上把 Android NDK、GCC、麒麟交叉编译器、方舟编译器全装上，靠环境变量切换或者包一层脚本来选平台。

听着简单，实际跑起来到处打架。PATH、CC、CXX、AR 这些环境变量被不同工具链来回改写；libstdc++.so、libm.so 这些同名库因为版本不一样，链接的时候报错；pthread.h、errno.h 这种头文件在不同平台下的定义不一样，预处理器经常串台。结果是同一份源码换个环境就编出不一样的结果，有时候直接编不过。我们曾经为这个问题维护过一套切换脚本，越写越复杂，最后没人敢动。

**（二）虚拟机隔离**

给每个平台开一个 VirtualBox 或 VMware 虚拟机。隔离是彻底了，但虚拟机太重——启动要几分钟，吃内存吃磁盘，镜像管理也麻烦。我们试过在 CI 上用虚拟机跑四平台编译，排队排到怀疑人生。

**（三）各平台原生构建**

Android 用 Gradle（Android.bp / build.gradle），Linux 用 Make，麒麟用自研的"开明包"构建系统，鸿蒙用 Hvigor。四套构建系统，四套配置语法，四套依赖声明方式。同一个源文件加进来，得在四个地方改；编译选项调一下，得对四处同步。维护成本高得离谱。

**（四）源码级条件编译（#ifdef）**

用宏在源码里塞平台分支。能解决一部分问题，但代码会被 `#ifdef` 撑得很丑，而且它解决不了"环境隔离"和"依赖库版本"这两个更底层的问题。

### 4.2 我们检索到的相关专利

下面这几篇是和我们的方向比较接近的公开专利，代理人写背景技术时可以引用并指出它们的局限。Google Patents 的公开链接也附上了。

**CN121187594B —— 一种基于 docker 实现 AGV 交叉编译的方法。**

这篇的做法是：x86 主机上用 qemu-aarch64-static + debootstrap 搭一个 ARM 根文件系统镜像，挂进 Docker 容器，然后写 CMake 工具链文件指向容器里的 ARM 根文件系统，再用 Makefile 封装编译命令，镜像靠远程仓库 ID 比对自动更新。用在 AGV（自动导引车）的 ARM 嵌入式导航工程上。

它的问题在于：只面向**单一目标架构（ARM）**和**单一 Linux 系**，没碰 Android、麒麟、鸿蒙这些；也没解决跨平台依赖库版本一致性的问题。我们做的是把四个异构操作系统放进来统一管，这是它没做的。

公开源：https://patents.google.com/patent/CN121187594B/zh

**CN111309451B —— 一种实现 Docker 镜像交叉编译的方法。**

它在 Docker 镜像交叉编译环境里装 qemu-user-static，靠 QEMU 在宿主机上模拟目标架构，实现跨架构的镜像构建。

问题在于它靠 QEMU 做指令级模拟，性能开销大；而且它解决的是"镜像能跑起来"，不是"源码怎么统一编译组织"，也没碰多操作系统和依赖库一致性。

公开源：https://patents.google.com/patent/CN111309451B/zh

**CN103853589A —— 一种跨平台的系统编译构建方法。**

用 CMake 做跨平台编译：先初始化环境变量和编译配置，上传 CMake 脚本，CMake 解析后用**当前平台自带的编译器**编译本平台产物，再打包发布。Windows 下生成工程文件，Linux 下生成 makefile。

注意：它用的是本平台编译器编本平台产物，属于**本地编译而不是交叉编译**，也没有容器隔离。只覆盖 Windows/Linux 两类。和我们差得比较远。

公开源：https://patents.google.com/patent/CN103853589A/zh

**CN111240685A —— 一种跨平台应用程序的构建方法及系统。**

在配置文件里列出所有平台名和 CPU 架构，排列组合得到多个应用环境，对各环境分别交叉编译，产物打包上传云端。用在 Go 语言 PC 端应用的多平台发布。

问题：没有容器隔离（环境冲突没解决），也没有统一构建组织（还是靠各平台原生工具链），没碰国产操作系统和依赖库一致性。

公开源：https://patents.google.com/patent/CN111240685A/zh

### 4.3 我们的方案跟它们到底差在哪

上面这些专利各有侧重，但有几个共同的缺口：要么只做单一架构、单一系统（CN121187594B、CN111309451B），要么不做交叉编译、不做隔离（CN103853589A、CN111240685A），而且**没有一个**建立起覆盖 Android、Linux、麒麟、鸿蒙这四类异构操作系统的统一交叉编译体系，也**没有一个**提出跨平台依赖库版本一致性管控的机制。

我们的本质区别，请代理人写"发明内容"和"创造性"时抓住这几条：

1. 我们是给四类**异构操作系统**分别建物理隔离的容器化交叉编译环境，不是单架构模拟，也不是多工具链混装在一台机器上；
2. 我们用一个 CMake 项目组织文件 + 工具链文件注入来统一组织四平台编译，不是各平台原生构建工具各搞各的；
3. 我们用"统一源码版本 + 统一编译选项 + SHA256 校验"来保证四平台系统依赖库行为一致——这一条，据我们检索，上面的专利都没提过；
4. 我们把麒麟国密、鸿蒙方舟和 Android、Linux 放在同一套统一构建体系下，而不是给国产系统单独适配。

## 五、我们具体是怎么做的（技术方案）

### 5.1 整个系统由四个模块组成

我们这套跨平台统一构建系统，拆开来看有四块：

**第一块：容器化交叉编译环境管理模块。** 给 Android、Linux、麒麟、鸿蒙各建一个独立的容器，每个容器里预装对应平台的编译工具链、链接器、标准库、系统头文件。容器之间靠容器运行时（Docker / containerd）把文件系统、进程、网络命名空间隔开。这一块解决的是"环境冲突"。

**第二块：CMake 统一构建组织模块。** 用 CMake 当跨平台构建工具，一份 CMake 项目组织文件描述源码编译逻辑，靠工具链文件注入和平台条件判断来适配各平台差异。这一块解决的是"构建系统碎片化"。

**第三块：统一版本系统依赖库构建模块。** 每个平台容器里的系统依赖库，都用同一个源码版本、同一组编译选项来构建，靠版本锁定文件、统一编译脚本和 SHA256 校验把各平台版本对齐。这一块解决的是"依赖库行为不一致"。

**第四块：容器编排与构建执行模块。** 用 Docker Compose 或 Kubernetes 把四个容器的编译任务并行调度起来，做到一键触发、并行编译、产物隔离输出。这一块解决的是"CI/CD 集成"和"执行效率"。

下面分开说每一块的具体做法。

### 5.2 容器化交叉编译环境管理模块

我们给四个平台各做了一个镜像，镜像内部是分层的。这个分层不是随便分的，是有讲究的——下面会说。

**（一）Android 编译容器**

以官方 Android NDK 镜像做底子，里面装 Clang/LLVM 工具链、lld 链接器、libc++ 和 Bionic C 库。容器里的环境变量（CC、CXX、CMAKE_TOOLCHAIN_FILE 之类）只在容器内部生效，不会泄漏到宿主机。

镜像我们分了四层来组织：
- 基础工具层：cmake、ninja-build、python3 这些通用构建工具；
- 工具链层：Android NDK；
- 依赖库层：预编好的统一版本第三方库；
- CMake 工具链文件层：内置 android.toolchain.cmake。

这样分层的好处是，依赖库层和工具链层可以独立缓存，改业务代码的时候不用重建依赖库。

**（二）Linux 编译容器**

以 Debian/Ubuntu 或 Alpine 做底，装 GCC/Clang 工具链、glibc 或 musl C 库、标准 GNU 链接器。支持 x86_64、ARM64 这些架构，编译器路径和 sysroot 由 CMake 工具链文件配。

**（三）麒麟编译容器**

这块是花了我们不少功夫的。以麒麟操作系统基础镜像做底，装适配国产 CPU 架构的交叉编译器——麒麟要支持龙芯 LoongArch、飞腾 ARM64、鲲鹏 ARM64 这三种。另外还装了麒麟安全增强库（KSAF）、开明包构建工具和国密算法支持库。

容器里集成了一个麒麟专用的 CMake 工具链文件 kylin.toolchain.cmake，里面配好了安全增强编译选项（比如 `-fstack-protector-strong`、`-D_FORTIFY_SOURCE=2`）和国密 SSL 库的路径。这些选项是麒麟合规要求强制的，别的地方不能随便关。

**（四）鸿蒙编译容器**

以 OpenHarmony 官方 DevEco Docker 镜像做底，装方舟编译器（ArkCompiler）、方舟运行时库、Hvigor 构建工具、鸿蒙 SDK。容器里配好 harmonyos.toolchain.cmake 工具链文件，对接方舟编译器的调用约定和链接规则。

方舟编译器和标准 GCC/Clang 的 ABI 不完全一样，链接规则也有自己的门道，所以这个工具链文件不是简单设几个变量就行的，我们花了一些时间摸清楚它的坑。

每个容器都有独立的文件系统、进程、网络命名空间，四个平台的工具链、系统库、头文件在物理层面被彻底隔开。本地多工具链共存那种"环境变量互相覆盖"的问题，在我们这里根本不存在——因为它们压根不在同一个进程空间里。

### 5.3 CMake 统一构建组织模块

我们选 CMake 3.28 及以上版本。为什么是 CMake？因为它原生支持工具链文件注入、原生支持平台检测变量，社区里几乎每个平台都有人写过工具链文件，四平台都能接进来。我们试过自己写一套元构建系统，后来发现没必要——CMake 配合工具链文件就够用了，少造一个轮子。

做到"一份 CMakeLists.txt 适配四平台"，靠的是三个机制：

**（一）平台自动检测**

CMake 自带的 `CMAKE_SYSTEM_NAME` 变量能认出 Android、Linux、HarmonyOS（鸿蒙官方已经贡献了对应的 system name）。麒麟没有标准的 system name，我们自己加了个 `KYLIN` 变量来扩展识别。

CMakeLists.txt 里用条件分支处理各平台差异，比如：

```cmake
if(ANDROID)
    # Android 特定的源文件、编译宏、链接库
elseif(CMAKE_SYSTEM_NAME STREQUAL "Linux")
    if(KYLIN)
        # 麒麟特有的国密适配
    else()
        # 通用 Linux
    endif()
elseif(CMAKE_SYSTEM_NAME STREQUAL "HarmonyOS" OR HARMONYOS)
    # 鸿蒙方舟适配
endif()
```

**（二）工具链文件注入**

每个容器内置对应平台的 CMake 工具链文件，通过 `-DCMAKE_TOOLCHAIN_FILE=...` 注入。注入之后，编译器、链接器、系统库路径、查找路径全都自动配好。

这里有个关键点：工具链文件里我们把查找路径**限制在容器内部**：

```cmake
set(CMAKE_FIND_ROOT_PATH_MODE_PROGRAM NEVER)
set(CMAKE_FIND_ROOT_PATH_MODE_LIBRARY ONLY)
set(CMAKE_FIND_ROOT_PATH_MODE_INCLUDE ONLY)
```

这样 CMake 找库的时候不会跑到宿主机的系统目录去，保证用的是容器里那一份统一版本的库，宿主机环境也不会被污染。

**（三）统一目标定义**

所有平台共享同一套 `add_executable()`、`add_library()` 目标定义，平台差异只落在工具链配置和条件编译选项上。编译标准（C11 / C++17）和输出目录结构也统一，编出来的产物组织方式是一致的。

### 5.4 统一版本系统依赖库构建模块（这是我们觉得最有价值的一块）

这一块说实话是我们花心思最多的地方，也是我们觉得最能支撑创造性的地方。

同一个加密逻辑，在 Android 上跑 BoringSSL，在 Linux 上跑 OpenSSL 3.0，在麒麟上强制走国密 SSL，在鸿蒙上跑自研安全模块——四套实现，行为对不齐。线程行为也对不齐：Android Bionic 的 pthread 和 Linux glibc 的 NPTL 在线程取消、信号处理上有差异。这种"平台特异性 bug"非常难查，因为它只在某个平台才暴露。

我们的思路是：**别在业务源码里去适配这些差异，而是在依赖库构建阶段就把差异抹平。** 具体靠四个机制：

**（一）依赖库源码锁定**

四个平台用的是**同一套依赖库源码包**——比如统一用 `openssl-3.0.8.tar.gz`、`curl-8.4.0.tar.gz`、`zlib-1.3.tar.gz`、`sqlite-3.44.0.tar.gz`。版本号和 SHA256 校验值统一写在一个 `dependency_versions.lock` 文件里：

```
openssl=3.0.8|sha256=...
curl=8.4.0|sha256=...
zlib=1.3|sha256=...
sqlite=3.44.0|sha256=...
```

各平台容器构建依赖库时，读的是同一份锁定文件，下载源码后再用 SHA256 校验一遍，保证四个平台拿到的是同一份源码。

**（二）统一编译选项**

每个依赖库都用同一组 CMake 编译配置：

```
-DCMAKE_BUILD_TYPE=Release       # 优化级别统一
-DBUILD_SHARED_LIBS=OFF          # 统一静态链接
-DCMAKE_POSITION_INDEPENDENT_CODE=ON   # 位置无关代码统一
```

这样静态/动态库属性、优化级别、代码生成属性就跨平台对齐了。

**（三）平台适配层注入**

平台特定的差异，我们不写进业务源码，而是放在**依赖库编译阶段**，靠注入适配补丁或者编译选项来解决。举几个实际例子：

- 麒麟：注入国密适配补丁 `-DKYLIN_GMSSL=1 -DPATCH_GMSSL=/opt/patches/gmssl.patch`；
- 鸿蒙：禁用它不支持的 POSIX 功能 `-DHARMONYOS_NO_POSIX=1 -DUSE_ARK_COMPAT=1`；
- Android：启用 BoringSSL 兼容层 `-DOPENSSL_NO_ENGINE=1`。

这样做的好处是业务源码保持干净，平台差异对业务开发者透明。

**（四）依赖库镜像缓存**

编好的依赖库我们打成容器镜像层或者独立的 tar 包，通过 Docker Registry 或私有镜像仓库分发。后续编译直接复用缓存镜像，不用每次都从头构建依赖库。这一步对 CI/CD 的构建效率提升很明显。

### 5.5 容器编排与构建执行模块

**（一）并行编译调度**

用 Docker Compose 或 Kubernetes 一次性把四个容器拉起来，各编各的，多核 CPU 能吃满。比起一个一个串行编，速度快很多。

**（二）源码只读挂载**

统一源码仓库通过容器卷（Volume）以**只读**方式挂进各容器。编译过程中源码改不了，这保证四个平台编的是同一份源码快照——不会出现"Android 容器编的时候源码是一个版本，轮到鸿蒙编的时候源码已经被改了"这种情况。

**（三）产物隔离输出**

各容器的编译产物落到各自的宿主目录：`build-android/`、`build-linux/`、`build-kylin/`、`build-harmonyos/`，互不覆盖。

**（四）一键触发**

一条 `docker-compose up` 就能触发四平台并行编译，不用手动切环境，也不用分多步操作。

## 六、这样做带来了什么好处（有益效果）

这部分请代理人在写"有益效果"时参考。我们用实际项目的数据来支撑：

1. **环境冲突彻底没了。** 四套工具链分别装在隔离容器里，环境变量、库文件、头文件互不覆盖。同一台开发机能同时跑四套环境。和本地多工具链共存比，这类冲突在我们这套方案里基本不会再出现。

2. **构建维护成本明显下降。** 构建工具收敛成 CMake 一种。原来 Android.bp、Hvigor 配置、开明包配置这些要分别维护，现在一份 CMakeLists.txt 覆盖四平台。我们估算构建配置的维护工作量降了六成以上。

3. **依赖库行为跨平台一致。** 四平台依赖库同源、同版本、同编译选项，运行时调的是行为一致的 API。以前那种"库版本差异引起的、只在某个平台才出现的 bug"基本消失了。

4. **国产操作系统适配门槛降下来了。** 麒麟、鸿蒙的容器化编译环境和 CMake 工具链文件我们都预先配好了。现在把一个 Linux/Android 项目迁到麒麟或鸿蒙，只要加几处平台条件分支和少量适配代码，不用重搭编译环境，也不用从头学新的构建工具。我们的实际项目里，适配周期缩短了一半以上。

5. **CI/CD 接得上。** 容器直接对接 Docker Registry、Kubernetes 这些云原生设施。和虚拟机比，容器启动是秒级而不是分钟级，资源占用小，镜像分层缓存也高效，扛得住 CI/CD 的高频构建。

6. **团队协作省心。** 同一份镜像通过私有仓库发给每个成员，大家用的编译环境完全一致。"在我机器上能编"这种事不会再扯皮了。

### 实施例 1：金融级加密 SDK 的跨平台编译

这个 SDK 要同时支持 Android 移动设备、Linux 桌面终端、麒麟服务器、鸿蒙嵌入式设备。

**用老办法的痛点：** Android 用 BoringSSL、Linux 用 OpenSSL 3.0、麒麟强制国密 SSL、鸿蒙没有 OpenSSL 支持，同一套加密逻辑要维护四套编译配置；四套工具链装一台机器上环境变量打架；四个构建系统（Android.bp / CMake / Hvigor / 开明包）配置维护困难。

**我们的做法：**

**步骤 1 —— 准备统一源码和 CMake 项目文件。** 项目目录结构如下，平台特定的代码放在各自的子目录里：

```
project/
├── CMakeLists.txt                # 统一构建文件
├── src/
│   ├── crypto_engine.cpp         # 平台无关加密逻辑
│   ├── android/sec_provider.cpp  # Android 特定
│   ├── kylin/gmssl_wrapper.cpp   # 麒麟国密适配
│   ├── harmonyos/ark_crypto.cpp  # 鸿蒙方舟适配
│   └── linux/openssl_default.cpp # Linux 标准 OpenSSL
└── dependency_versions.lock      # 统一版本锁定
```

**步骤 2 —— 构建容器化编译环境。** Android 容器的 Dockerfile 大致是这样（有删减）：

```dockerfile
FROM ubuntu:22.04

# 基础层：通用构建工具
RUN apt-get update && apt-get install -y cmake ninja-build python3 git wget

# 工具链层：Android NDK
ENV ANDROID_NDK=/opt/android-ndk-r26b
COPY android-ndk-r26b-linux.zip /tmp/
RUN unzip /tmp/android-ndk-r26b-linux.zip -d /opt/

# 依赖库层：统一版本依赖库
COPY third_party /opt/third_party
RUN cd /opt/third_party && ./build_dependencies.sh android

# CMake 工具链文件层
COPY android.toolchain.cmake /opt/cmake/

# 环境隔离：仅容器内生效
ENV CC=$ANDROID_NDK/toolchains/llvm/prebuilt/linux-x86_64/bin/aarch64-linux-android21-clang
ENV CXX=$ANDROID_NDK/toolchains/llvm/prebuilt/linux-x86_64/bin/aarch64-linux-android21-clang++
ENV CMAKE_TOOLCHAIN_FILE=/opt/cmake/android.toolchain.cmake
```

麒麟容器类似，但基础镜像用 `kylin-server:v10`，装国产 CPU 交叉编译器（`gcc-loongarch64-linux-gnu` 等）和麒麟安全增强库（`kylin-sec-lib`、`kylin-gmssl`）。鸿蒙容器用 `openharmony/deveco:latest`，方舟编译器和 SDK 已预装。

**步骤 3 —— 统一 CMakeLists.txt 的关键片段。** 平台自动检测 + 统一依赖查找 + 平台特定适配层：

```cmake
cmake_minimum_required(VERSION 3.28)
project(CryptoSDK VERSION 1.0.0 LANGUAGES C CXX)

# 平台自动检测
if(ANDROID)
    message(STATUS "Building for Android, API level ${ANDROID_NATIVE_API_LEVEL}")
elseif(CMAKE_SYSTEM_NAME STREQUAL "Linux")
    if(KYLIN)
        message(STATUS "Building for Kylin OS, arch ${KYLIN_ARCH}")
    else()
        message(STATUS "Building for Generic Linux")
    endif()
elseif(CMAKE_SYSTEM_NAME STREQUAL "HarmonyOS" OR HARMONYOS)
    message(STATUS "Building for HarmonyOS")
endif()

# 统一编译标准
set(CMAKE_C_STANDARD 11)
set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CXX_STANDARD_REQUIRED ON)

# 统一依赖库查找
find_package(OpenSSL REQUIRED)
find_package(CURL REQUIRED)
find_package(ZLIB REQUIRED)
find_package(SQLite3 REQUIRED)

# 平台无关业务源码
add_library(crypto_sdk STATIC src/crypto_engine.cpp src/aes_gcm.cpp ...)

# 平台特定适配层
if(ANDROID)
    target_sources(crypto_sdk PRIVATE src/android/sec_provider.cpp)
    target_compile_definitions(crypto_sdk PRIVATE PLATFORM_ANDROID USE_BORINGSSL_COMPAT)
elseif(KYLIN)
    target_sources(crypto_sdk PRIVATE src/kylin/gmssl_wrapper.cpp)
    target_compile_definitions(crypto_sdk PRIVATE PLATFORM_KYLIN USE_GMSSL)
    target_link_libraries(crypto_sdk kylin_gmssl)
elseif(HARMONYOS)
    target_sources(crypto_sdk PRIVATE src/harmonyos/ark_crypto.cpp)
    target_compile_definitions(crypto_sdk PRIVATE PLATFORM_HARMONYOS USE_ARK_CRYPTO)
    target_link_libraries(crypto_sdk ark_crypto)
else()
    target_sources(crypto_sdk PRIVATE src/linux/openssl_default.cpp)
endif()
```

**步骤 4 —— 统一依赖库构建脚本。** 各平台读同一份锁定文件，SHA256 校验源码，用同一组 CMake 选项编译，只是平台特定补丁不一样：

```bash
#!/bin/bash
# build_dependencies.sh
PLATFORM=$1
OPENSSL_VERSION=$(grep "^openssl=" dependency_versions.lock | cut -d= -f2)
# ... 其他版本

build_library() {
    local name=$1 version=$2 flags=$3
    sha256sum -c "${name}-${version}.tar.gz.sha256" || exit 1   # 校验
    # 统一 CMake 配置
    cmake -B build -DCMAKE_BUILD_TYPE=Release -DBUILD_SHARED_LIBS=OFF \
          -DCMAKE_POSITION_INDEPENDENT_CODE=ON $flags
    cmake --build build --parallel $(nproc)
    cmake --install build
}

case $PLATFORM in
    android)  build_library openssl $OPENSSL_VERSION "... -DOPENSSL_NO_ENGINE=1" ;;
    kylin)    build_library openssl $OPENSSL_VERSION "... -DKYLIN_GMSSL=1 -DPATCH_GMSSL=/opt/patches/gmssl.patch" ;;
    harmonyos) build_library openssl $OPENSSL_VERSION "... -DHARMONYOS_NO_POSIX=1 -DUSE_ARK_COMPAT=1" ;;
    linux)    build_library openssl $OPENSSL_VERSION "..." ;;
esac
```

**步骤 5 —— 容器编排并行编译。** Docker Compose 把源码以只读方式挂进四个容器，各编各的，产物落到各自目录：

```yaml
services:
  build-android:
    image: cross-compile-android:latest
    volumes:
      - ./src:/workspace/src:ro
      - ./CMakeLists.txt:/workspace/CMakeLists.txt:ro
      - ./build-android:/workspace/build
    command: sh -c "cmake -B build -DCMAKE_TOOLCHAIN_FILE=/opt/cmake/android.toolchain.cmake && cmake --build build --parallel $(nproc)"
  # build-linux / build-kylin / build-harmonyos 同理
```

**步骤 6 —— 一键执行。** `docker-compose up build-android build-linux build-kylin build-harmonyos`，四个平台并行编译，输出四个静态库。我们验证过：用 `strings` 查四个产物，里面都是 `OpenSSL 3.0.8`，版本确实一致。

### 实施例 2：国产 CPU 多架构适配

一款工业控制软件要适配麒麟的三种国产 CPU（龙芯 LoongArch、飞腾 ARM64、鲲鹏 ARM64），同时支持 Android 和鸿蒙。

麒麟容器里装多架构交叉编译器，给每种架构单独构建依赖库目录。然后用 CMakePresets.json 管理多架构预设——每个架构一个 preset，指向各自的工具链文件和输出目录。实际编译时 `cmake --preset=kylin-loongarch64` 这样一条条触发，产物落到各自目录。这套思路也能扩展到更多架构。

### 实施例 3：接入 GitLab CI/CD

把四平台编译做成 CI 流水线的 build 阶段，四个 job 并行跑，各自 build + test，最后 package 阶段把四个平台的产物归档成一个跨平台 SDK 压缩包。代码提交后自动触发，产物作为 artifact 留存。因为是容器，job 启动快、资源省，扛得住高频提交。

## 八、关于权利要求，我们的一些想法（请代理人参考）

下面是我们对权利要求布局的初步想法，供代理人定稿时参考。具体怎么分独立权利要求和从属权利要求，请代理人按审查实务把握。

我们认为独立权利要求至少应该覆盖这几个特征：

1. 为 Android、Linux、麒麟、鸿蒙**分别构建独立的容器化交叉编译环境**，每个容器预装对应平台的编译工具链、链接器、标准库及系统头文件，容器之间通过容器运行时实现文件系统、进程、网络命名空间隔离；
2. 采用 **CMake 作为统一的跨平台构建工具**，通过同一套 CMake 项目组织文件描述源码编译逻辑，借助工具链文件注入和平台条件判断适配各平台；
3. 为各平台容器构建**统一版本、一致编译选项的系统依赖库**，依赖库来源于同一源码包，通过统一编译脚本和校验机制（SHA256）保证版本一致性；
4. 通过容器编排工具**并行执行**各平台编译任务，输出对应平台的目标产物。

从属权利要求可以考虑围绕下面这些点展开（代理人也请补充我们没想到的）：

- 镜像分层结构（基础工具层 / 工具链层 / 依赖库层 / 工具链文件层）；
- 工具链文件中查找路径限制（`CMAKE_FIND_ROOT_PATH_MODE_*`）只限定在容器内；
- 麒麟容器的安全增强编译选项和国密 SSL 库适配；
- 鸿蒙容器对方舟编译器调用约定和链接规则的适配；
- 依赖库镜像缓存与私有仓库分发机制；
- 源码只读挂载保证多平台基于同一源码快照；
- 产物按平台隔离输出目录；
- CMakePresets 管理多 CPU 架构预设；
- CI/CD 流水线集成方式。

## 九、我们觉得哪些是真正的创新点（请代理人重点保护）

写在最后，是想跟代理人交个底——在我们看来，这套方案里哪些是别人没做过、值得我们重点保护的：

**第一，"四类异构操作系统 + 容器物理隔离"这个组合。** 现有的容器化交叉编译专利基本都只做单一架构或单一 Linux 系。把麒麟国密、鸿蒙方舟这种国产系统，和 Android、Linux 放进同一套容器化统一构建体系，据我们检索没人做过。

**第二，"统一源码版本 + 统一编译选项 + SHA256 校验"的依赖库一致性机制。** 这是解决"同一逻辑在不同平台行为不一致"这个老大难问题的办法。别的专利要么没碰依赖库一致性，要么只做单平台。这一条我们觉得创造性最强。

**第三，"平台差异下沉到依赖库编译阶段"的做法。** 不在业务源码里做平台适配，而是把差异封装在依赖库构建的适配补丁和编译选项里。这让业务源码保持干净，也使得适配可以独立演进。

**第四，CMake 工具链文件注入 + 查找路径限定在容器内。** 这个机制同时解决了"环境隔离"和"依赖库版本锁定"两个问题——工具链文件既配好了编译器，又把查找路径锁在容器内，保证不会误用宿主机或别的平台的库。

请代理人在撰写时，把这几个点作为创造性的核心论据，和背景技术里那几篇对比专利的差异呼应起来。

### 10.5 统一依赖库构建脚本（完整）

```bash
#!/bin/bash
# build_dependencies.sh --- 统一构建各平台依赖库

PLATFORM=$1   # android | linux | kylin | harmonyos
VERSION_FILE="dependency_versions.lock"

OPENSSL_VERSION=$(grep "^openssl=" $VERSION_FILE | cut -d= -f2)
CURL_VERSION=$(grep "^curl=" $VERSION_FILE | cut -d= -f2)
ZLIB_VERSION=$(grep "^zlib=" $VERSION_FILE | cut -d= -f2)
SQLITE_VERSION=$(grep "^sqlite=" $VERSION_FILE | cut -d= -f2)

INSTALL_DIR="/opt/third_party/${PLATFORM}"

build_library() {
    local name=$1 version=$2 source_tarball="${1}-${2}.tar.gz" flags=$3
    sha256sum -c "${source_tarball}.sha256" || exit 1
    tar xzf $source_tarball && cd ${name}-${version}
    cmake -B build -DCMAKE_BUILD_TYPE=Release -DBUILD_SHARED_LIBS=OFF \
          -DCMAKE_POSITION_INDEPENDENT_CODE=ON \
          -DCMAKE_INSTALL_PREFIX=${INSTALL_DIR} ${flags}
    cmake --build build --parallel $(nproc)
    cmake --install build
    cd ..
}

case $PLATFORM in
    android)
        COMMON_FLAGS="-DCMAKE_TOOLCHAIN_FILE=/opt/cmake/android.toolchain.cmake -DANDROID_NATIVE_API_LEVEL=21"
        build_library "openssl" $OPENSSL_VERSION "$COMMON_FLAGS -DOPENSSL_NO_ENGINE=1" ;;
    kylin)
        COMMON_FLAGS="-DCMAKE_TOOLCHAIN_FILE=/opt/cmake/kylin.toolchain.cmake"
        build_library "openssl" $OPENSSL_VERSION "$COMMON_FLAGS -DKYLIN_GMSSL=1 -DPATCH_GMSSL=/opt/patches/gmssl.patch" ;;
    harmonyos)
        COMMON_FLAGS="-DCMAKE_TOOLCHAIN_FILE=/opt/cmake/harmonyos.toolchain.cmake"
        build_library "openssl" $OPENSSL_VERSION "$COMMON_FLAGS -DHARMONYOS_NO_POSIX=1 -DUSE_ARK_COMPAT=1" ;;
    linux)
        build_library "openssl" $OPENSSL_VERSION "" ;;
esac

build_library "curl" $CURL_VERSION "$COMMON_FLAGS -DCURL_USE_OPENSSL=ON"
build_library "zlib" $ZLIB_VERSION "$COMMON_FLAGS"
build_library "sqlite" $SQLITE_VERSION "$COMMON_FLAGS"
```
